# The Paraloom deliverable system — two codebases, one contract

Status: 2026-06-17 (v2 — collapsed from three systems to two: the standalone `paraloom-deliverables` React repo was migrated into Paraloom itself and is retired. The deliverable domain deployed to prod 2026-06-15; by-reference manifest push and the human-readable evidence layer were added 2026-06-17). This doc is the current-state map; the `History` section below carries the rationale.

## The two systems

| System | Repo | Stack | Role |
|---|---|---|---|
| **Paraloom Agent** | `~/Documents/Code/paraloom-agent` (this repo) | Claude Code workflows + Python tools | **Produces** deliverables. Runs experiments end-to-end (`/exp-research` → `/exp-build` → `/exp-review`), writes the article + reviews + evidence, distills everything into the manifest, pushes it to Paraloom (`/exp-deliver`), and applies client answers as revisions (`/exp-revise`). |
| **Paraloom** | `~/Herd/paraloom` (canonical checkout; local serve via `php artisan serve` at `127.0.0.1:8002`, not Herd's server) | Laravel 12 + Vue 3 SPA (app.paraloom.ai) + a separately-mounted React/Plate.js client app | **Owns AND presents** deliverables. System of record: stores manifests as versioned `Deliverable` records attached to teams/experiments, materializes action items, collects client answers/comments/attestations, keeps the activity audit ledger, sends client email — and serves the client-facing experience itself at `/d/<token>` (React app vendored at `resources/deliverables/`, mounted via a blade view + route above the Vue catch-all). |

## How data flows

```
 (1) produce                          (2) store + collect + present
paraloom-agent ────MCP tools────►  paraloom (Laravel + in-repo React client)
      ▲                                  │
      │                                  ├── email link (Mailables) ──► client opens /d/<token>?p=<magic>
      └────────MCP polling──────────────┘◄── answers / comments / attestations / opt-outs
 (3) revise: agent applies answers through the methodology, re-runs the
     review battery, pushes a new revision; the cycle repeats until green.
```

1. **Agent → Paraloom (MCP, write):** `create-deliverable`, `push-deliverable-revision`, `set-deliverable-status`, `send-deliverable`. The agent authenticates to Paraloom's existing MCP server (`app/Mcp`, `mcp:write` scope) — the same surface it already uses for campaigns and experiments. While the deliverable domain is local-only, `/exp-deliver mode:'local'` uses `php artisan deliverables:ingest {path} --team= --experiment=` instead.
2. **Paraloom → client (same-origin):** Laravel emails the link (Mailables, help@paraloom.ai); the React app at `/d/<token>` reads `GET /api/public/deliverables/{token}` and writes answers/attestations/comments/opt-outs back to the same token-scoped endpoints — all same-origin now, no CORS dependency, no separate deploy. Participants are identified by per-person magic tokens (`?p=`), never accounts.
3. **Paraloom → agent (MCP, read):** the agent polls `get-deliverable-activity` (a `/loop` cron) for new answers; `/exp-revise` consumes them — answers are inputs to the methodology, never raw edit commands — re-runs the battery, and pushes the next revision.

## The contracts (what each boundary agrees on)

- **The manifest (`deliverable.json`)** — the unit of exchange: article + opportunity header + granular checklist + quote-anchored action items + dossier. Produced by `tools/build-deliverable-manifest.py` (which hard-validates anchors/enums/status before anything leaves the agent), stored verbatim by Paraloom as a revision snapshot, rendered by the in-repo React client. Quote-based anchors (`{quote, prefix?, suffix?, section?}`) are what make the manifest portable.
  - **Human-readable evidence layer (added 2026-06-17).** Each dossier doc keeps its raw working-file markdown unchanged (the audit trail) and may carry an optional plain-language `summary`; the manifest may also carry a top-level `evidenceOverview` (the featured "why we're confident" narrative). Both are generated by the `/exp-review` synthesize step into `manifest-inputs.json` (`dossierSummaries` keyed by dossier filename + `evidenceOverview`), attached by the manifest builder, and rendered by the client's Evidence tab (`summary` first, raw markdown behind a "view the full technical record" toggle). Both optional, so pre-existing deliverables render unchanged.
- **`DELIVERABLES-API.md`** (paraloom repo root) — the authoritative endpoint + tool + manifest reference, including the in-repo client-app section. On a contract conflict, fix the mismatch, don't silently adapt. Note: the public API returns snake_case keys; the React client normalizes (`src/api/normalize.ts`).

## Ownership boundaries (who may change what)

- Article content changes **only** in the agent (through the methodology + review battery). Paraloom stores and renders read-only.
- Client input (answers, comments, attestations, opt-outs) is written **only** through Paraloom's public token API and is never edited after the fact (append-only activity ledger).
- Status advances only when the gates allow: the agent re-runs the battery before `ready`; humans resolve `attestation` items; the FI sign-off and the publish remain human acts.
- Sends are gated: `send-deliverable` refuses while `requires_send_approval` is set (Ryan's explicit OK for early sends).
- **Approved ≠ ready to publish (added 2026-06-18).** A client may approve at any time, even while a requested change is still pending on our side; approval is their go-ahead, not a readiness signal. "Ready to publish" is *derived* server-side (`Deliverable::readyToPublish()`: no unresolved blocking action items + pre-publish checks cleared) and surfaced as `ready_to_publish` on the public payload and the MCP `get-deliverable`/`list-deliverables` output. **A `fact-confirm` whose client answer is a *correction* (not "Confirmed correct as written.") stays unresolved until the agent applies it** — resolution is `DeliverableActionItem::isResolved()` (status `applied`/`returned`, or answered-and-not-an-outstanding-correction). The agent resolves a correction by pushing a revision that edits the article **and** marks that item `status: applied`; the ingest preserves the client's answer text. The publish gate lives at `set-deliverable-status` (refuses `published` unless approved **and** ready), not at the client approve step. Full contract in the paraloom repo `docs/deliverables/README.md` → "Readiness vs approval".

## Operational notes & known failure modes

- **Inline-manifest push can silently drop the dossier (2026-06-12, deliverable 5 / southern-wi-home-equity).** On the MCP push path, a subagent reads `deliverable.json` and emits the manifest as a tool argument. The `dossier` array is large (~90KB of markdown across ~9 docs), and an LLM asked to echo a payload that big inline will abbreviate it — the push agent sent a manifest with `[manifestVersion, deliverable, opportunity, checklist, actionItems]` and **no `dossier`**, shipping an empty evidence tab. This is **not** a Paraloom bug: `DeliverableIngestService::applyManifest` stores the manifest verbatim, the schema declares `dossier`, and the React client renders it — Paraloom stored exactly what it was handed. The loss is at the agent→tool boundary.
- **The artisan/file path is byte-exact.** `php artisan deliverables:ingest {file} --team=` reads the manifest with `json_decode` and passes the full array — no LLM re-typing — so the dossier survives. This is why `/exp-deliver mode:'local'` is the safe local path, and the manual repair is a re-ingest of the same `deliverable.json` (slug-idempotent → same `deliverable_id` + token).
- **`/exp-deliver` now self-guards (Reconcile phase).** Verify reports `servedDossierCount` vs `manifestDossierCount`. If the MCP push came up short: on the local server (`mcp__paraloom-local__`) it auto re-ingests via artisan and re-verifies; on a remote/prod server it **halts** (`dossier-dropped-on-push`) and does **not** notify the client, since it cannot ingest into a remote DB from here. The push prompt also instructs the agent to pass the manifest verbatim and never omit `dossier`.
- **By-reference manifest push (implemented 2026-06-17 — the structural fix).** `create-deliverable` / `push-deliverable-revision` now accept a **`manifest_ref`** as an alternative to the inline `manifest`. The agent stages the manifest over plain HTTP first (`POST {apiBase}/api/deliverable-manifests`, Sanctum bearer auth, the file sent with `curl --data-binary @deliverable.json` — no model in the byte path), gets back a short-lived `ref` (cache-staged, scoped to the uploading user, ~30 min TTL, single-use), and passes that `ref` to the MCP tool instead of the ~90KB payload. The MCP tool resolves the ref and calls the same `DeliverableIngestService::store()`. This removes the inline-echo failure mode entirely while keeping every deliverable WRITE an MCP tool call. Both `/exp-deliver` and `/exp-revise` use this on the `mcp` (prod) path; `mode:'local'` still uses the byte-exact artisan ingest. Server code: `StagedManifestController`, `App\Mcp\Traits\ResolvesManifestRef`, the two tools, `tests/Feature/Deliverable/ManifestByReferenceTest.php` (paraloom repo, branch `deliverable-manifest-by-reference`).
  - **Auth/secret:** the agent reads `PARALOOM_API_TOKEN` from `<repo>/.paraloom.env` (preferred) or `~/.paraloom.env` (gitignored via `.*.env`). Mint it on the target env with `php artisan mcp:create-token <your-email>` (a Sanctum PAT with `['*']`). The PAT **must belong to the same Paraloom user the MCP connector authenticates as**, or the staged ref is rejected as foreign. For the headless poller, configure the prod MCP connector as a bearer-token HTTP server (mirroring the `paraloom-local` entry in `~/.claude.json`) using that same PAT, so the `push-deliverable-revision` call also works in cron.
  - The **Reconcile phase remains** as a belt-and-suspenders check (Verify still compares `servedDossierCount` vs `manifestDossierCount`); with by-reference it should always match.
- **A deliverable is only auto-revised if its experiment folder records it (2026-06-18).** The poller's `discover()` (`tools/poller-tick.py`) yields `(slug, token)` only for experiments whose `experiment.md` has a `## Deliverable` section with a token; everything else is invisible to the loop. A deliverable created outside the `/exp-deliver` flow (e.g. the MetriFi smoke test, prod deliverable 4) has no such folder, so client answers on it sit **unactioned** — that gap was how the approved-with-an-open-correction bug surfaced. When you create a deliverable you intend the loop to service, give it an experiment folder with a `## Deliverable` section (token + ids), as `/exp-deliver` does.

## Lifecycle of one deliverable (end to end)

experiment runs (agent, prod) → draft + battery green-enough → manifest built + validated (agent) → `create-deliverable` / artisan ingest (Paraloom) → email with link (Paraloom) → client answers at `/d/<token>` (Paraloom) → poller picks it up (agent) → `/exp-revise` applies + re-reviews + pushes revision (agent → Paraloom) → repeat until all blocking items closed + battery green → compliance attestation in-app (human) → `scheduled` (+5 business days, opt-out window) → publish (CMS, human) → experiment flips `published`, measurement starts → `/exp-measure` results eventually land on the same deliverable page.

## History

Until 2026-06-12 this was a three-system architecture with a standalone React SPA (`paraloom-deliverables` repo) as the client surface. Ryan collapsed it: the React app was vendored into the paraloom repo (`resources/deliverables/`, one package.json, one Vite config with Vue + React entries, Tailwind sources scoped per app, `@deliverables` alias) and the standalone repo is retired. The old `~/Documents/Code/paraloom` checkout was never canonical and has been deleted; `~/Herd/paraloom` is the real one.
